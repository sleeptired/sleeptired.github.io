### **2. 수정을 해야 한다면, 기존 코드의 수정을 최소화 할 수 있는 방안이 있을까?**
1) AlchemyWorkshop,PotionRecipe 파일 분할  

> **수정 사항**  
> ● 각 클래스 헤더파일에는 클래스 선언부만 남겨 인터페이스를 명확히 함  
> ● 각 클래스 cpp파일에는 세부 구현 로직을 분리하여 컴파일 속도 및 가독성 향상    
{: .prompt-info }

2) PotionRecipe 멤버 변수 캡슐화  
PotionRecipe 내 데이터가 외부에서 무분별하게 수정되는 것을 방지하기 위해 public 접근 제어자를 private으로 변경하고 캡슐화를 적용했습니다.  

```diff
 class PotionRecipe {
- public:
-    std::string potionName;
-    std::vector<std::string> ingredients;
+ private:
+    std::string potionName;
+    std::vector<std::string> ingredients;

+ public:
+    PotionRecipe(const std::string& name, const std::vector<std::string>& ingredients)
+        : potionName(name), ingredients(ingredients) {
+    }
 };
```
   
### **3. 객체지향 프로그래밍 언어라면 SOLID 원칙을 잘 준수하고 있나?**

1) SRP (단일 책임 원칙)
PotionRecipe: 레시피 데이터 정보만 전문적으로 담습니다.  
AlchemyWorkshop: 레시피들의 목록을 관리하는 로직에만 집중합니다.  

2) OCP (개방-폐쇄 원칙) 점검 및 해결

> **문제점**  
> 만약 PotionRecipe에 gold, time 같은 새로운 변수가 추가된다면?  
> ● addRecipe 함수의 매개변수를 매번 뜯어고쳐야 합니다.  
> ● 해당 함수를 호출하던 프로젝트 내의 **모든 코드**에서 에러가 발생하며 일일이 수정해야 합니다.  
{: .prompt-danger }

```diff
// AlchemyWorkshop.h
- void addRecipe(string name, vector<string> ingredients); 
+ // 완성된 객체 자체를 넘겨줌으로써 확장에 유연하게 대처
+ void addRecipe(const PotionRecipe& newRecipe);
```

> **해결 결과**  
> ● 유지보수성 향상: PotionRecipe 클래스에 어떤 변수가 추가되더라도 AlchemyWorkshop의 코드는 단 한 줄도 수정할 필요가 없습니다.  
> ● 코드 안정성: 유저의 입력을 받아 객체를 생성하는 main 함수 코드만 수정하면 되므로, 기존 매니저 클래스의 로직을 건드릴 위험이 사라집니다.   
{: .prompt-tip }
